# ComfyUI_AdvancedHandRefiner

ComfyUI向けカスタムノード集。手指のinpaint/生成結果を解剖学的に正しく補正するための3ノードで構成されます。

## インストール

1. このリポジトリを ComfyUI の `custom_nodes/` 配下にクローン（またはZIP展開）
2. 依存関係をインストール

```bash
# ComfyUI portable版(Windows)の例
python_embeded\python.exe -m pip install -r ComfyUI\custom_nodes\ComfyUI_AdvancedHandRefiner\requirements.txt

# venv/conda等で構築している場合
pip install -r requirements.txt
```

`torch` / `numpy` / `opencv-python` は通常ComfyUI本体が既に提供しているため、
`requirements.txt` には含めていません。詳細は [`requirements.txt`](./requirements.txt) のコメントを参照してください。

3. ComfyUIを再起動すると、`HandRefiner` カテゴリに3ノードが追加されます

## ノード一覧

| ノード | 表示名 | 役割 |
|---|---|---|
| `AdvancedHandOrientationOptimizer` | 👋 Hand Orientation & Crop Optimizer | 手の向き検出・回転正規化・クロップ |
| `AdvancedHandMaskRefiner` | ✨ Advanced Anatomical Mask Refiner | 指の輪郭強調・手首境界のぼかし |
| `AdvancedHandSeamlessStitcher` | 🪡 Seamless Stitch & Color Matcher | 逆変換・シームレス合成 |
| `AdvancedHandQualityChecker` | 🔍 Advanced Hand Quality Checker | 手の解剖学的妥当性を自動判定（指の欠損・癒着・過剰等を検出） |
| `AdvancedHandAutoFixer` | 🔁 Advanced Hand Auto Fixer | 検出→クロップ→インペイント→品質チェック→リトライを自動的に繰り返す |

## 想定ワークフロー

```
[元画像]
   ↓
[OrientationOptimizer] → cropped_image, remap_info
   ↓                          ↓
[何らかのInpaintノード]        │
   ↓                          │
[MaskRefiner] ← マスク         │
   ↓                          │
[SeamlessStitcher] ← original_image, remap_info
   ↓
[最終画像]
```

### 複数の手を処理したい場合

`process_all_hands`（`AdvancedHandOrientationOptimizer`のパラメータ）を
`True`にすると、**ノードを複製せずに1系統だけで**、画像内で検出された
全ての手をまとめてバッチ処理できます（既存の「複数画像バッチ処理」の
仕組みをそのまま応用しています）。

```
[元画像] → [OrientationOptimizer(process_all_hands=True)] → cropped_image(バッチ), remap_info(リスト)
                                                                    ↓
                                    [何らかのInpaintノード] → [MaskRefiner] → [Stitcher] → [最終画像]
```

`AdvancedHandMaskRefiner`・`AdvancedHandSeamlessStitcher`はどちらも
バッチ処理に対応済みのため、そのまま繋ぐだけで全ての手が処理されます
（`Stitcher`の`original_image`には元の1枚の画像をそのまま渡せば、
検出された手の数だけ自動的に使い回されます）。

`process_all_hands=False`（既定は`True`。個別に指定した場合）は従来通り
`hand_index`で指定した1つの手のみを処理します。手ごとにパラメータ
（`hand_index`）を分けて個別にワークフローを組みたい場合は、以下の
ようにノードチェーンを手の数だけ複製することもできます（1回目は
`hand_index=0`、2回目は
`hand_index=1`、…）。

```
[元画像] ──┬─→ [OrientationOptimizer(hand_index=0)] → ... → [Stitcher] ──┐
           └─→ [OrientationOptimizer(hand_index=1)] → ... → [Stitcher] ←─┘
                                                        (1回目の出力をoriginal_imageに使う)
```

検出処理（YOLO+MediaPipe+SAM2）自体は`hand_index`に依存せず同じ結果に
なるため、**同一画像・同一パラメータでの検出結果はプロセス内でキャッシュ**
されます。そのため、上記のようにノードチェーンを複製しても、検出処理が
手の数だけ重複して実行されることはありません（2回目以降はキャッシュを
再利用するため、実質的に追加コストはほぼゼロです）。

## 検出パイプライン

手の検出は `YOLO（バウンディングボックス） → MediaPipe（骨格ランドマーク） → SAM2（画素単位セグメンテーション）`
の3段階パイプラインで構成されており、各検出器は互いの結果を補完し合います。
複数の手が写っている場合、各検出器間の対応付けはbboxのIoU（Intersection over
Union）に基づいて行われるため、検出器ごとに手の順序が異なっていても正しく
統合されます。各検出器はモデルファイルが無い場合に自動的にスキップされるため、
`models/`配下の一部が欠けていてもクラッシュせず動作します（ただし機能は限定されます）。

- **YOLO** (`hand_yolov8s.pt`/`.onnx`, `Bingsu/adetailer`配布): 手の見逃しを減らすバウンディングボックス検出
- **MediaPipe** (`hand_landmarker.task`, Google公式): 手の向き・関節構造の把握
- **SAM2** (`sam2_hiera_tiny`, `vietanhdev/segment-anything-2-onnx-models`配布): 画素単位の精密セグメンテーション。
  デコーダの生出力は256×256固定解像度のため、大きな画像では自動的にタイル分割して
  推論することで実効解像度を向上させている（`sam2_tile_size`で調整可能）。
  タイル境界の誤検出を防ぐため、重なり領域は連続値のまま平均化してから閾値判定し、
  さらに小さな孤立領域を除去するクリーンアップを行う

## ノードパラメータ詳細

### 👋 Hand Orientation & Crop Optimizer (`AdvancedHandOrientationOptimizer`)

手の骨格向きを検出し、垂直になるよう回転・パディング付きでクロップします。
inpaintノードに渡す前段として使うことを想定しています。

| パラメータ | 型 | デフォルト | 説明 |
|---|---|---|---|
| `image` | IMAGE | (必須) | 入力画像 |
| `padding` | INT | 32（0〜256, step 8） | クロップ時に手の周囲へ追加する余白ピクセル数 |
| `min_detection_confidence` | FLOAT | 0.5（0.1〜1.0） | 検出パイプライン全体の最低信頼度しきい値 |
| `hand_index` | INT | 0（0〜19） | 複数の手が検出された場合に処理対象とする手のインデックス（0=最も信頼度が高い手）。範囲外の値は警告の上、最後の手にクランプされる |
| `detection_mode` | 選択式 | `full` | 検出パイプラインの実行モード。`full`=YOLO+MediaPipe+SAM2、`yolo_mediapipe`=SAM2を省略、`mediapipe_only`=MediaPipeのみ（最速）|
| `process_all_hands` | BOOLEAN | `True` | `True`にすると`hand_index`を無視し、検出された全ての手をバッチとしてまとめて処理する（ノードを複製せずに複数の手を扱える）。`False`にすると`hand_index`で指定した1本のみを処理する |

**出力**: `cropped_image`（回転・クロップ後の画像）, `remap_info`（`SeamlessStitcher`に渡す逆変換情報）

手が検出できなかった場合は、警告ログを出して入力画像をそのまま返します（クラッシュしません）。
バッチ入力に対応しています。バッチ内で検出した手ごとにクロップサイズが異なる場合は、
バッチ内最大サイズへ左上寄せでゼロパディングして1つのIMAGEテンソルにまとめます
（`remap_info`はバッチサイズ1の場合は単一dict、2以上の場合はdictのリストを返します）。

### ✨ Advanced Anatomical Mask Refiner (`AdvancedHandMaskRefiner`)

粗いマスク（inpaintノードの出力等）を、手の骨格情報に基づいて指の輪郭を強調し、
手首との境界をなめらかにぼかします。

| パラメータ | 型 | デフォルト | 説明 |
|---|---|---|---|
| `image` | IMAGE | (必須) | マスクに対応する画像 |
| `mask` | MASK | (必須) | 粗いマスク（inpaintノード等の出力） |
| `wrist_blur` | INT | 15（1〜99, step 2, 奇数のみ） | 手首境界をぼかすカーネルサイズ |
| `finger_sharpness` | FLOAT | 1.0（0.0〜5.0） | 指の輪郭強調の強さ（0で無効） |
| `min_detection_confidence` | FLOAT | 0.5（0.1〜1.0） | 検出パイプライン全体の最低信頼度しきい値 |
| `use_sam2_mask` | BOOLEAN | False | SAM2のセグメンテーションマスクを併用するか |
| `sam2_blend_strength` | FLOAT | 0.5（0.0〜1.0） | `use_sam2_mask=True`時のブレンド強度。0で粗いマスクのみ、1でSAM2マスク優先。両方が前景と判定した領域は強度に関わらず前景として維持されます |
| `hand_index` | INT | 0（0〜19） | 複数の手が検出された場合に処理対象とする手のインデックス（0=最も信頼度が高い手）。範囲外の値は警告の上、最後の手にクランプされる |
| `detection_mode` | 選択式 | `full` | 検出パイプラインの実行モード。`full`=YOLO+MediaPipe+SAM2、`yolo_mediapipe`=SAM2を省略、`mediapipe_only`=MediaPipeのみ（最速）|
| `sam2_tile_size` | INT | 512（128〜2048, step 64） | `use_sam2_mask=True`時、この値を超える画像はタイル分割して推論する（実効解像度が向上するが処理時間は増加。小さくするほど高精度・低速）|

**出力**: `refined_mask`（補正後マスク）

手が検出できなかった場合は、入力マスクをそのまま返します。`use_sam2_mask=True`でも
検出パイプラインにSAM2が含まれていない/セグメンテーションに失敗した場合は、
粗いマスクにフォールバックします。このノードはバッチ入力に対応しており、
バッチ内の各画像・マスクを独立して処理します。

### 🪡 Seamless Stitch & Color Matcher (`AdvancedHandSeamlessStitcher`)

`OrientationOptimizer`で行った回転・クロップを逆変換し、補正済みマスクの領域だけを
元画像に自然に合成します。境界付近の色調も自動でマッチングします。

| パラメータ | 型 | デフォルト | 説明 |
|---|---|---|---|
| `original_image` | IMAGE | (必須) | `OrientationOptimizer`への入力画像（合成先） |
| `inpainted_image` | IMAGE | (必須) | inpaint等で生成された画像（`cropped_image`と同サイズ） |
| `refined_mask` | MASK | (必須) | `MaskRefiner`の出力マスク |
| `remap_info` | REMAP_INFO | (必須) | `OrientationOptimizer`の出力（逆変換情報） |
| `color_match_strength` | FLOAT | 0.8（0.0〜1.0） | 境界周辺の色調マッチングの強さ（0で無効） |

**出力**: `final_image`（合成後の最終画像）

`remap_info`が`OrientationOptimizer`からのdictのリスト（バッチ処理時）であれば、
バッチの各要素を正しく対応付けて処理します。`remap_info`が単一dict（バッチサイズ1で
生成されたもの）の場合、`original_image`側にバッチサイズ>1の画像を渡しても、
警告ログを出したうえで先頭画像のみを処理します（単一dictでは1画像分の
逆変換情報しか表現できないため）。

## 既知の制約・推奨ワークフロー

### 全身画像・イラスト調画像での検出精度について（実写データによる検証結果、2026-07-07）

YOLO/MediaPipeの手検出モデルはいずれも**実写の手**を主な学習データとしているため、
以下の条件が重なると検出精度が大きく低下し、服のシワや髪の毛など手以外の部分を
誤って「手」として検出してしまうことがあります（コード上の不具合ではなく、
検出モデル自体の適用範囲の限界です）。

- **画像全体に対して手が小さく写っている**（全身画像・複数人物が写っている構図等）
- **アニメ調・イラスト調の絵柄**（実写と陰影・輪郭線・質感が大きく異なる）
- 指を握り込む等、**特殊なポーズで指の形が分かりにくい**

このような場合、`detection_mode`を変えても改善しないことが多く（`mediapipe_only`
単体でも検出0件になる、あるいは`full`で低信頼度のまま誤検出したものを拾って
しまう等）、`min_detection_confidence`の調整だけでは解決しないことがあります。

#### 推奨ワークフロー: 事前クロップ

**手が写っている部分を大まかに（`ImageCrop`等の標準ノードで）事前に切り出してから
`OrientationOptimizer`に渡す**運用を強く推奨します。全身画像のまま渡すと
「小さく写った手に対して検出を試みて失敗する」ことになりますが、手を画面に
対して十分大きく切り出しておくだけで、検出精度が大きく改善するケースが
多いことを実写データで確認済みです（ちょうどADetailer等が顔や手を検出して
自動クロップする2段階方式と同じ考え方です）。

```
[Load Image(全身/イラスト)]
     ↓
[ImageCrop等で手周辺を大まかに切り出し]
     ↓
[👋 OrientationOptimizer] → ...(以降は通常のワークフロー)
```

### 指が深く絡み合った（インターレースした）ポーズについて（実写データによる検証結果、2026-07-07）

上記はイラスト調画像特有の制約でしたが、**実写真であっても**、以下のような
構図では検出が完全に失敗する（YOLO・MediaPipe双方とも検出数0）ことを
実データで確認しています。

- **両手の指を深く組み合わせた（編み込むように重なり合った）ポーズ**
  （例: 祈るように指を絡めて組んだ手を至近距離で写した構図）
- 手が画面全体を埋め尽くすような極端なクローズアップ構図

この場合、`min_detection_confidence`を`0.1`まで下げても検出できず、
`detection_mode`の変更でも改善しません。さらに、bboxを手動指定して
SAM2単体（検出をバイパス）で試しても、指の絡み合いが複雑すぎて
手の塊全体を正しくセグメンテーションできない（一部分しか検出でき
ない）ことも確認しています。

**現状の結論**: これはYOLO/MediaPipe/SAM2いずれのモデルにとっても
根本的に難しいケースであり、コード上の調整では解決しません。指が
はっきり分離して見える、比較的シンプルな構図の画像でお試しいただく
ことを推奨します。現在のパイプラインには、bbox/マスクを手動指定して
検出を完全にバイパスする機能は無いため、将来的な検討事項としています。

### 🔍 Advanced Hand Quality Checker（実験的機能）

検出された手が解剖学的に妥当か（指の欠損・癒着・過剰な指等が無いか）を
自動判定します。Phase 6で開発した3つの指標を組み合わせて判定します：

- 凸包の凹みベースのマスク解析（指の欠損・強い癒着の検出に強い）
- 骨格化ベースのマスク解析（際どい間隔で挿入された余分な指の検出に強い）
- MediaPipeランドマークの関節妥当性チェック（指を握り込んだ/曲げた
  ポーズに対してマスクベースより頑健）

**入力**: `image`

**パラメータ**: `min_detection_confidence`, `hand_index`, `detection_mode`,
`process_all_hands`（他ノードと共通）に加え、`expected_fingers`
（本来あるべき指の本数、デフォルト5）

**出力**: `is_abnormal`（BOOLEAN、異常の疑いがあれば`True`）、
`quality_report`（STRING、手ごとの詳細判定結果）

`is_abnormal`の出力は、後続のワークフローで「崩れている疑いがある
場合のみinpaintし直す」といった条件分岐に利用できます。

⚠️ **既知の限界**: 指を握り込んだ/曲げたポーズでは、「余分な指の疑い」
判定が誤検知することがあります（マスクベースの手法が握り込みポーズを
苦手とするため。この判定はMediaPipeの構造上ランドマークで代替できず、
現時点でも未解決です）。また、この機能は実データでの検証途上であり、
今後も精度改善を継続する予定です。詳細はMILESTONES.mdを参照してください。

**手の向きによる「欠損」誤判定への対応（2026-07-09、ユーザー指摘により
発見・修正）**: 手の向き（カメラに対する傾き）によっては、2〜3本の指
しか画面上にはっきり見えない構図（指が奥行き方向を向いている等）が
解剖学的に正常なケースがあります。従来はランドマークの2D投影座標のみで
判定していたため、こうした「向きによる自然な短縮」を「指の欠損」と
誤判定し、正常な手を5本指に描き直してしまう可能性がありました。
MediaPipeが出力するz座標（奥行き）を活用し、指の長さ判定を向きに
依存しない3D距離で行うよう修正済みです。ただし、以下の限界は残ります:

- 「余分な指」の判定（マスクベースのみ）には効果が及びません
- ランドマークが取得できない場合のマスクベースのフォールバック判定
  （見た目のシルエットに写っている指の本数を数える手法）には、
  そもそも画像に写っていない指を検出する原理的な手段がないため、
  この対策は適用されません
- MediaPipeのz座標自体も推定値であり、x/yほど精度が高くない場合が
  あります

### 🔁 Advanced Hand Auto Fixer（実験的機能）

検出→クロップ→インペイント→品質チェック→（必要なら）リトライを、
1つのノードで自動的に繰り返します。将来目標「不完全な手を見つけ、
描画し直す」を実現する中核ノードです。内部でComfyUI本体のサンプリング
機構（`VAEEncodeForInpaint`→KSampler相当→`VAEDecode`という、標準的な
「Detailer」系ノードと同じ構成）を呼び出すため、他のノードとは異なり
`model`, `positive`, `negative`, `vae` の入力が必要です。

**入力**: `image`, `model`, `positive`, `negative`, `vae`

⚠️ **複数の手が写っている画像で一部の手しか直らない場合**: `process_all_hands`
は既定で`True`のため、**検出された手は全て自動的に処理対象**になります
（`hand_index`は無視されます）。特定の1本だけを処理したい場合は
`process_all_hands`を`False`にし、`hand_index`で対象を指定してください。
なお検出自体（YOLO+MediaPipe+SAM2の統合結果）は`process_all_hands`の
設定に関わらず全ての手を正しく検出しています。片方の手だけ検出されて
いない（そもそも検出0件になる）ように見える場合は、実際にその手が
画像上に描かれているかどうか（袖に隠れて実質未描画になっていないか等）
を先に確認することをお勧めします。

**主なパラメータ**: `seed`, `steps`, `cfg`, `sampler_name`, `scheduler`,
`denoise`（通常のKSamplerと同様。`sampler_name`/`scheduler`はComfyUI標準の
KSampler系ノードと同じ選択式ドロップダウンで、`comfy.samplers`から
取得した実際の一覧が表示される）、`max_retries`（最大リトライ回数、
デフォルト3）、`max_crop_dimension`（クロップの一辺の絶対的な上限px、
デフォルト768。検出結果が悪化して異常に広い範囲を検出してしまった
場合の安全弁）。加えて他ノードと共通の`hand_index`/`detection_mode`/
`process_all_hands`/`expected_fingers`にも対応しています。

**サンプリング系パラメータの推奨値（既定値）**:

| パラメータ | 既定値 | 理由 |
|---|---|---|
| `sampler_name` | `dpmpp_2m` | 指の輪郭のような細部の再現性を重視した、detailer系ワークフローで速度と精細さのバランスが良いとされる定番sampler |
| `scheduler` | `karras` | dpmpp系samplerと組み合わせた際にノイズスケジュールが滑らかになり、細部の描画精度が安定しやすい標準的な選択 |
| `steps` | `25` | 手は指の分離・爪の形状など細部が崩れやすいため、一般的なKSamplerの既定（20前後）よりやや高めに設定 |
| `cfg` | `7.0` | 多くのモデルで破綻が少ない標準的な値 |
| `denoise` | `0.6` | 元の構図・周囲との一貫性を保ちつつ、破綻した指を作り直すのに十分な強さ |
| `mask_grow_pixels` | `6` | インペイント境界を軽く拡張し、継ぎ目を目立ちにくくする |

より高精度な結果を優先し処理時間が長くても良い場合は、`sampler_name`を
`dpmpp_2m_sde`や`dpmpp_3m_sde`に、より高速に試したい場合は`euler`に
変更することを推奨します（いずれもドロップダウンから選択できます）。

**出力**: `image`（最終結果）、`fix_report`（STRING、手ごとの試行回数・
結果の詳細）

リトライのたびに`seed`をインクリメントしながら再生成し、
`AdvancedHandQualityChecker`と同じ判定ロジックで「問題なし」と
判定されるか、`max_retries`回に達するまで自動的に繰り返します。

⚠️ **重要な注意（テスト範囲の限界）**: 開発環境に実際の拡散モデル・
GPUが無いため、このノードが内部で呼び出すサンプリング機構
（`common_ksampler`等）は、**実際のComfyUI環境（本物のモデル・VAE）での
エンドツーエンドの動作確認ができていません**。検出→クロップ→品質判定→
リトライという制御フロー自体はモックを使った単体テストで厳密に
検証済みですが、実際のサンプリング統合部分（正しいマスク成長・
latentの扱い等）はユーザー環境での動作確認・フィードバックをお願い
したい部分です。

#### リトライ間のクロップサイズ肥大化対策（実写環境での実行ログにより発見・修正済み、2026-07-09）

実際にComfyUI環境で実行したユーザーから提供された実行ログの調査により、
**リトライを重ねるたびにサンプリング対象の画像サイズが際限なく拡大し、
1回目の試行と比べて3回目の試行では約50倍も処理時間が悪化する**、という
重大なパフォーマンス問題を発見・修正しました。

**原因**: リトライのたびに「貼り戻し後の（まだ完璧ではない）画像」に
対して手を再検出し、その検出結果（landmarks）を使って次のクロップ範囲を
決め直す設計になっていました。もし再検出結果が悪化する（インペイントの
継ぎ目やアーティファクトを誤って「手」の一部と判定してしまう等）と、
クロップ範囲＝サンプリング対象の画像サイズが際限なく肥大化し、
KSamplerの1ステップあたりの処理時間が指数的に悪化していました。

**修正**: 1回目の試行で得られたクロップサイズを上限として記憶し、
2回目以降のリトライではその上限を超えないようクロップ範囲を制限する
ようにしました（`utils/geometry.py`の`compute_padded_bbox`に
`max_width`/`max_height`パラメータを追加）。これにより、再検出結果が
悪化した場合でも、サンプリングコストが際限なく跳ね上がることはなく
なります。回帰テストも追加済みです。

#### クロップサイズの絶対的な上限を追加（VAEデコード中のクラッシュ対策、2026-07-11）

上記の対策は「同じ手の2回目以降の試行」でのクロップ肥大化は防げましたが、
**(a) 1回目の試行自体がそもそも検出結果の悪化により異常に大きくなる
ケース、(b) `process_all_hands=True`で複数の手を処理する際、後続の手の
「1回目の試行」が独自に大きくなるケース**には対応できていませんでした。
実際にユーザー環境で、この種の巨大クロップによりVAEデコード中に
VRAM枯渇由来と見られるネイティブクラッシュ
（`Fatal Python error: Aborted`、`comfy/ldm/models/autoencoder.py`の
`decode()`内で発生）が発生することを実行ログで確認しました。

また、クロップが手に対して過度に大きいと、拡散モデルが「周囲の生地
（グローブ等）の継続」を優先してしまい、**手そのものがほとんど描かれず
肌がほぼ見えない**という別の症状（ユーザー報告あり）にもつながることが
分かりました。

**修正**: 試行回数・手の処理順序に関わらず常に効く「絶対的な上限」
`max_crop_dimension`（`AdvancedHandAutoFixer`の新規パラメータ、
デフォルト768px）を追加し、1回目の試行から一貫して適用するように
しました。修正前のコードで新規テストを実行すると、2000px四方の画像で
検出結果が画像全体近くに広がった場合、1回目の試行でクロップが2779px
（キャンバスの回転パディング込みでキャンバスサイズ自体を超える）まで
肥大化することを確認した上で、修正後は上限（768px）以内に収まることを
検証しました。

⚠️ **続報（2026-07-11）**: 上記の修正を適用した状態で再現テストして
いただいたところ、**同じクラッシュ・同じ速度の低下パターンが再現**
しました。実行ログのスタックトレースの行番号を確認したところ、修正後の
コードが実際に使われていることは確認できています。つまり、クロップ
サイズの上限だけでは根本原因を捉えきれていない可能性が高いと判断し、
以下の対応を追加しました:

- `_run_inpaint_sampling`に、実際に使用されるクロップサイズを明示する
  診断ログ（`HandAutoFixer: インペイント実行 crop=WxH ...`）を追加。
  次回のログ取得で、クロップサイズの上限が実際に効いているかどうかを
  確定できるようにしました
- 各試行の開始時に、どの画像・どの手・何回目の試行かを明示するログ
  （`HandAutoFixer: [image_index=.., hand=.., attempt=../..] ...`）を
  追加し、ワークフロー上の他ノードのログ出力と区別できるようにしました
- VAEデコード直前に`torch.cuda.empty_cache()`を呼び、サンプリング中に
  断片化した可能性のあるVRAMを整理してからデコードに臨むようにしました
  （デコードに必要な連続領域の確保失敗によるクラッシュのリスク低減）

**正直な現状**: クロップサイズが原因ではない場合、低速化の実態は
ComfyUI本体のVRAM管理（VRAMが逼迫した際にモデルの重みをレイヤーごとに
GPU⇔CPU間でストリーミングする、いわゆる`--lowvram`/`--novram`相当の
動作）である可能性も否定できません。この場合、当プラグイン側で
根本的に解消することは難しく、ユーザー環境のVRAM容量・ComfyUIの
起動オプションの見直しが必要になる可能性があります。次回のログで
クロップサイズが確定次第、追って調査します。

✅ **根本原因を確定・修正しました（2026-07-11）**: 上記の診断ログを
追加した状態でユーザーに再現していただいた結果、決定的な証拠が得られ
ました。3回目の試行のログに以下が記録されていました:

```
[INFO] HandAutoFixer: [image_index=0, hand=0, attempt=3/4] クロップ処理開始 (crop上限=238x269)
[WARNING] HandOrientationOptimizer: 手が検出できませんでした(image_index=0)。入力画像をそのまま返します。
[INFO] HandAutoFixer: インペイント実行 crop=2304x3456 (padding後=2304x3456)
```

`crop上限=238x269`が正しく設定されているにも関わらず、実際に使われた
クロップは`2304x3456`（元画像のフルサイズそのもの）でした。

**真の原因**: `_crop_for_hand`には、`selected.landmarks is None`
（`Sam2HandDetector`がlandmarksを信頼できないと判断し、bboxのみの
結果にフォールバックした場合に発生する）の際、**`max_crop_size`を
完全に無視して元画像を丸ごとそのまま返してしまう**という別のバグが
存在していました。これまでの2回の修正（相対上限・絶対上限）は、いずれも
「landmarksが取得できている通常の検出結果」に対してのみ機能する経路
だったため、この「landmarksが無い場合の全画像フォールバック」経路には
一切効いていませんでした。

**修正**: landmarksが無くても`bbox`があれば、bbox+paddingでクロップ
する（`max_crop_size`も正しく適用）よう変更しました。bboxも無い
（本当に手がかりが一切無い）場合も、`max_crop_size`が指定されていれば
画像中央基準でその上限まで縮小してから返すようにし、無制限の巨大画像が
後段のサンプリングに渡ることが無いようにしました。`max_crop_size`が
未指定の場合（`AdvancedHandOrientationOptimizer`単体の手動ワークフロー
等）は、従来通り元画像全体を返す後方互換動作のままです。

修正前のコードで新規テストを実行すると、実際にクロップの高さが
2000px（キャンバスサイズそのもの、`max_crop_dimension=512`を大幅に
超過）になることを確認した上で、修正後は上限内に収まることを検証
しました。

#### SAM2推論中の "Windows fatal exception: access violation" クラッシュ対策（2026-07-09）

実行ログの提供により、SAM2のタイル分割推論中に、まれに
`onnxruntime`が発生させるネイティブレベルのクラッシュ
（`access violation`）が発生することが判明しました。

**原因**: タイル分割推論（大きな画像を複数タイルに分けて並列処理する
最適化）は、各タイルのエンコード/デコードを`ThreadPoolExecutor`で
並列実行していましたが、これは「onnxruntimeの`InferenceSession.run()`は
異なるスレッドから同一セッションを並行して呼び出しても安全」という
前提に基づいていました。しかし実際には、実行プロバイダ（特にWindows環境
で使われることの多いDirectML等）によってはこの前提が成り立たず、複数
スレッドが同時に同一セッションの`run()`を呼び出した際にネイティブ
クラッシュを引き起こすことが、実際のユーザー環境のクラッシュログで
確認されました。

**修正**: エンコーダ・デコーダそれぞれの実際の`session.run()`呼び出しを
`threading.Lock`で直列化するようにしました。タイルごとの画像切り出し・
前処理・後処理はスレッドプールのまま並列に実行されるため、並列化の
効果は完全には失われません（ネイティブ推論呼び出し自体のみを安全側に
倒しています）。修正前のコードで新規テストを実行すると実際に複数
スレッドが同時に`run()`内部へ入ってしまう（`max_concurrent=6`）ことを
確認した上で、修正後は常に1スレッドずつ（`max_concurrent=1`）に
なることを検証しています。

#### CUDA実行時のonnxruntimeログについて（2026-07-11）

GPU（CUDA実行プロバイダ）環境の実行ログには、SAM2推論時に以下のような
`onnxruntime`由来の警告が表示されることがあります。それぞれの意味と
対応状況をまとめます。

- `Error merging shape info for output... Falling back to lenient merge.`
  → SAM2エンコーダのONNXモデルファイル自体に、一部の中間出力の形状
  情報が静的に確定していない箇所があることに起因する警告です。
  onnxruntimeが実行時の実際の形状を採用してそのまま処理を続けるため、
  **無害**です（モデルファイル側の変換時メタデータの問題であり、
  こちら側のコードで対応できるものではありません）。
- `N Memcpy nodes are added to the graph... negative impact on performance`
  → モデル内の一部の演算がCUDA実行プロバイダに未対応で、CPUへ
  フォールバックするためGPU⇔CPU間のデータコピーが発生している、
  という**性能に関する情報**です。動作の正しさには影響しません。
- `ScatterND with reduction=='none' only guarantees to be correct if
  indices are not duplicated.`
  → SAM2デコーダ内部が、bbox隅・MediaPipeランドマーク等の座標
  プロンプトをposition embeddingへ変換する際にScatterND演算を
  使っており、**渡した座標に完全な重複があると、CUDA上ではスレッド間の
  書き込み順序が保証されず結果が非決定的になりうる**、という警告です。
  bboxが極端に小さい（実質1点に潰れている）場合や、境界条件で
  landmarks側の点がbboxの隅と厳密に一致する場合に、この重複が
  発生しうることを確認したため、デコーダへ座標を渡す直前に完全一致する
  重複座標を除去する`_dedupe_points_and_labels()`を追加しました
  （`utils/sam2_inference.py`）。ただし、これは「呼び出し側が渡す座標
  同士の重複」のみを解消するものであり、SAM2デコーダ内部が使う離散
  グリッドの解像度までは制御できないため、「異なるが非常に近い」座標が
  モデル内部でさらに衝突する可能性はモデルアーキテクチャに起因する
  限界として残ります。
